# Verification Report — account-access

_Generated by `/verify-req` (security demo)._

## Requirement
Let a user log in and view their account. Security: a user may view ONLY their
own account; passwords must never be returned; wrong/empty logins rejected.

## Result
20 tests — 15 pass, 4 fail, 1 manual (Node `node:test`). Code does NOT satisfy
the security requirements.

| behavior | what it checks | result |
|---|---|---|
| B1–B6 | login succeeds with right creds, rejected otherwise | PASS |
| B7 | user can view their own account | PASS |
| **B8** | **a user must NOT view another user's account** | **FAIL (IDOR)** |
| **B9** | **password must never be returned to the caller** | **FAIL (leak)** |
| B10 | password never shown anywhere (logs/UI/errors) | MANUAL |

## Root causes
- **B8 (access control / IDOR):** `getAccount(currentUserId, targetUserId)` ignores
  `currentUserId` — any logged-in user can read any account.
- **B9 (secret leak):** `login` and `getAccount` return the raw user object,
  including the plaintext `password`.

## npm-audit advisory
Skipped / not applicable — no `package.json` (single dependency-free file).

## Handoff (paste to the builder chat)
> Two security bugs in `src/auth.js`: (1) `getAccount` lets a user read someone
> else's account — enforce that `currentUserId` owns the account. (2) login and
> getAccount return the password — strip it before returning. Behaviors B8, B9.
